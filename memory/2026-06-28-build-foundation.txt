**CRITICAL BUG FOUND AND FIXED (Windows-first invariant, ADR-0007) — spawnClaude was fundamentally
broken on Windows and had been the WHOLE TIME**, including in the earlier analyze_footage "verification"
from the #24 session (that one got a response only by accident/inconsistent PATH state — do not trust
that as proof the CLI spawn ever reliably worked before this fix). Root cause, found via direct
reproduction (not guessing):
1. `@openvideo/render`'s resolveBin(name) tried extensions in order ["", ".exe", ".cmd", ".bat"] on
   win32. npm installs a global command as BOTH an extensionless POSIX shebang shim (e.g.
   `C:\Users\<u>\AppData\Roaming\npm\claude`, not a native PE executable) AND a `claude.cmd` shim. The
   extensionless one was found FIRST and returned as "resolved" — but Node's child_process.spawn()
   cannot execute it (ENOENT), because Windows CreateProcess needs a real PE binary. FIX: reordered to
   [".exe", ".cmd", ".bat", ""] in packages/render/src/execute.ts (extension-preferring order), with a
   comment explaining why order matters here specifically.
2. cli-adapter's spawnClaude didn't even call resolveBin — it spawned the bare string "claude" and let
   Node/Windows try to resolve it, which also failed (same ENOENT, or picked the wrong shim depending
   on shell state). Even after routing through a resolveBin-equivalent, spawning the found `claude.cmd`
   directly (no shell) throws EINVAL — Node refuses to exec .cmd/.bat without `shell:true` (a Node
   security hardening from ~v16+). Tried shell:true as a quick fix and it "worked" for --version but
   printed a Node deprecation warning: "Passing args to a child process with shell option true can lead
   to security vulnerabilities, as the arguments are not escaped, only concatenated" — UNACCEPTABLE
   here because SpawnOptions.prompt is arbitrary user text passed as an argv entry; shell:true would be
   a real command-injection vector the moment a user's prompt contains shell metacharacters. Rejected
   that approach even though it "worked."
3. REAL FIX (packages/cli-adapter/src/spawn.ts): `resolveClaudeBin(name)` now, on finding a `.cmd`
   shim, READS the shim's contents and follows it to its real target — npm's Windows shims are a fixed,
   well-known template ending in `"%dp0%\<relative-path>\claude.exe"  %*` (or a `.js` file for pure-JS
   CLIs). `%dp0%` is a batch variable meaning "this .cmd's own directory + trailing backslash" (set via
   `%~dp0`) — we expand it ourselves as `path.dirname(cmdPath) + path.sep`, then `path.normalize()` the
   result (raw concatenation produces a harmless but ugly double-backslash). If the target is a native
   `.exe`, spawn it directly. If it's a `.js`, spawn `[process.execPath, thatJsPath, ...realArgs]` —
   node.exe is itself a real PE binary, always safe to spawn with an argv array and NO shell. This is
   now exported as `resolveClaudeBin` (renamed from a private-ish idea) returning `{command, argsPrefix}`,
   consumed as `spawn(command, [...argsPrefix, ...buildClaudeArgs(o)], {cwd, stdio})` — no shell, ever.
   4 new tests incl. 2 that build a fake .cmd-shim-over-.exe and .cmd-shim-over-.js layout in a temp dir
   (win32-only, `{skip: process.platform !== "win32"}`) and assert the resolved target + argsPrefix.
